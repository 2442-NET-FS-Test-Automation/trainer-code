# Change Tracking, SaveChanges, and Data Seeding

## Learning Objectives
- Explain how EF Core tracks entity states and turns them into SQL at `SaveChanges()`.
- Name the entity states (Unchanged, Modified, Added, Deleted, Detached) and what moves an entity between
  them.
- Seed baseline data with `HasData` and explain why seed rows carry explicit primary keys.
- Reset known state with an idempotent endpoint instead of re-seeding.

## Why This Matters
With EF you rarely write an `UPDATE` statement. You load an `InventoryItem`, subtract from
`CurrentStock`, call `SaveChanges()` ‚Äî and the right SQL happens. That magic is the **change tracker**,
and it is also where the optimistic-concurrency story lives (the tracker is what notices a `RowVersion`
conflict ‚Äî see `efcore-concurrency.md`). Seeding matters for the same reason tests do: results only mean
something if every run starts from the same known data, so baseline rows ship in a migration and a small
endpoint restores them.

## The Concept

### Entity states
Every entity a `DbContext` knows about has a state in its change tracker:

| State | How it got there | SQL at SaveChanges |
|---|---|---|
| `Unchanged` | loaded by a query | none |
| `Modified` | you set a property on a tracked entity | `UPDATE` |
| `Added` | `db.Products.Add(entity)` | `INSERT` |
| `Deleted` | `db.Products.Remove(entity)` | `DELETE` |
| `Detached` | never tracked, or tracker cleared | none |

A small diagnostic endpoint makes the states visible (a learning device ‚Äî a real API would never publish
its change tracker):

```csharp
var unchanged = db.Products.First();          // queried -> Unchanged
var modified  = db.Products.Skip(1).First();  // still Unchanged...
modified.Price += 1;                          // ...now Modified - no method call needed

db.Products.Add(new Product { Sku = "BK-TMP", Name = "Tmp", Price = 1m });  // Added (not in DB yet!)

var states = db.ChangeTracker.Entries()
    .Select(e => new { entity = e.Entity.GetType().Name, state = e.State.ToString() })
    .ToList();

db.SaveChanges();          // one UPDATE + one INSERT, in one transaction
db.ChangeTracker.Clear();  // manual detach-everything (rarely needed in request code)
```

Two things trip people up:

- **Mutation is detection.** You do not call an "update" method; setting a property on a *tracked* entity
  is enough. EF compares current values against the snapshot it took at load time.
- **`Add` does not insert.** Nothing touches the database until `SaveChanges()`. Until then the new entity
  exists only in the tracker, with a temporary key.

### SaveChanges: the unit of work completes
`SaveChanges()` walks the tracker, generates one statement per Added/Modified/Deleted entity, and runs them
**in a single transaction** ‚Äî all of your session's changes commit or none do. That per-call atomicity is
exactly what one-operation-one-transaction designs hang on (fulfill an order: decrement stock + mark
fulfilled + write the event, atomically ‚Äî see `efcore-concurrency.md`). After a successful save, entities
return to `Unchanged` and identity keys generated by the database are written back onto your objects ‚Äî
which is why an order-seeding endpoint calls `SaveChanges()` once to get order ids, then uses those ids
for related rows and saves again.

### Seeding with HasData
Baseline data the app cannot run without (a starter catalog, its stock, a couple of customers) is seeded
in `OnModelCreating` and shipped *inside a migration*:

```csharp
b.Entity<Product>().HasData(
    new Product { Id = 1, Sku = "BK-001", Name = "Clean Code",               Price = 32.00m },
    new Product { Id = 2, Sku = "BK-002", Name = "The Pragmatic Programmer", Price = 38.00m },
    new Product { Id = 3, Sku = "BK-003", Name = "Refactoring",              Price = 45.00m });

b.Entity<InventoryItem>().HasData(
    new InventoryItem { Id = 1, ProductId = 1, CurrentStock = 5 },
    new InventoryItem { Id = 2, ProductId = 2, CurrentStock = 3 },
    new InventoryItem { Id = 3, ProductId = 3, CurrentStock = 8 });
```

Note the **explicit `Id` values**. `HasData` runs inside the migration, before the database can hand out
identity values, so seed rows must carry their own keys (and FKs). EF diffs seed data like schema: change a
seeded row and the next migration updates it. `HasData` plus a reset endpoint (below) is a common,
migration-native substitute for a bulk "POST /seed" import.

### Resetting state idempotently
Between test or benchmark runs, stock must return to the same baseline. A small endpoint that is safe to
call any number of times does it:

```csharp
app.MapPost("/seed/reset", (LibraryDbContext db) =>
{
    foreach (var inv in db.Inventory)
        inv.CurrentStock = inv.ProductId switch { 1 => 5, 2 => 3, 3 => 8, _ => inv.CurrentStock };
    db.SaveChanges();
    return Results.Ok("stock reset");
});
```

Detail worth copying: it switches on the FK scalar `inv.ProductId`, **not** `inv.Product.Id` ‚Äî the
`Product` navigation is not loaded here, so touching it throws a `NullReferenceException`. That is a
classic real-world bug shape: navigations are only objects if something loaded them. Navigation loading
rules are in `loading-strategies-raw-sql.md`.

## Say It in an Interview
- *"EF's change tracker gives every known entity a state ‚Äî Unchanged, Modified, Added, Deleted, Detached ‚Äî
  and `SaveChanges()` turns those states into INSERT/UPDATE/DELETE inside a single transaction."*
- *"You don't call an update method: setting a property on a tracked entity is enough, because EF diffs
  current values against the snapshot it took at load."*
- *"`Add` doesn't insert ‚Äî nothing hits the database until `SaveChanges()`; until then the entity lives
  only in the tracker with a temporary key."*
- *"`HasData` seeds baseline rows through migrations, with explicit primary keys because identity values
  don't exist at migration time; pair it with an idempotent reset endpoint for repeatable runs."*

## Check Yourself
1. You load an entity, set one property, and call `SaveChanges()`. What state transitions happened and
   what SQL ran?
2. Why do `HasData` rows need hard-coded primary keys?
3. `db.Orders.Add(order); var id = order.Id;` ‚Äî why is `id` useless here, and what fixes it?
4. `SaveChanges()` with three modified entities and one added ‚Äî how many transactions?
5. A reset endpoint reads `inv.Product.Id` and throws `NullReferenceException`, but "the FK is right
   there." Explain.

**Answers:** (1) Unchanged (on load) -> Modified (on property set); one `UPDATE` at save, then back to
Unchanged. (2) The seed executes inside the migration, before the database can generate identity values ‚Äî
EF also diffs seed rows across migrations by key. (3) `Add` only stages the entity in the tracker with a
temporary key; call `SaveChanges()` first, and the database-generated id is written back onto the object.
(4) One ‚Äî all statements from a `SaveChanges()` call commit or roll back together. (5) `Product` is a
navigation property, only populated if it was loaded (`Include` etc.); use the FK scalar
`inv.ProductId` instead.

## Summary
- The change tracker gives every known entity a state; property mutation on a tracked entity marks it
  Modified with no method call.
- `SaveChanges()` converts tracked states to INSERT/UPDATE/DELETE and runs them in one transaction ‚Äî the
  unit of work commits atomically.
- `HasData` seeds baseline rows through migrations, with explicit PKs because identity is not available at
  migration time; `HasData` + a reset endpoint is the migration-native seed pattern.
- Reset endpoints must be idempotent and must not touch unloaded navigation properties.
